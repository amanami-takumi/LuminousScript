# 黒画面問題の教訓

## 症状

- ニューゲーム開始後、Console には `Loading scene: 1-1` が出る
- BGM は再生される
- しかし画面は黒く見える
- DevTools では `title-content` に反応が残る

## 今回の本質

- `loadScene()` 自体は動いており、シーン遷移ロジックは止まっていなかった
- 問題は「タイトル画面が完全に消えていない可能性」と「単一HTMLで完結すべきなのに、`file:` 環境で余計な読み込み失敗が混ざる余地」があったこと

## 具体的な原因候補と教訓

### 1. 画面切り替えを `opacity` だけに頼ると、旧画面が DOM 上に残る

- `.screen` は非アクティブ時に `opacity: 0` になるだけだった
- この方式だと、見た目では消えていても DOM 上には残る
- DevTools で `title-content` が拾え続けるため、原因の切り分けを誤りやすい

### 教訓

- 画面切り替えは `opacity` だけでなく `display` または `hidden` も併用する
- 「見えない」と「存在しない」は別物として扱う
- タイトル画面のような全画面UIは、非表示時に物理的に外す設計のほうが安全

## 2. 単一HTMLが前提なら、相対ファイル参照を残してはいけない

- `favicon.ico` のような補助リソースでも、相対参照が残ると `file:` 実行時に不要な問題の種になる
- 本体が正常でも、Console に `file:` 系エラーが混ざると本筋の原因が見えにくくなる

### 教訓

- 「単一HTML完結」が要件なら、画像・音声だけでなく favicon なども含めて埋め込みを基本にする
- 相対パス参照が最終HTMLに残っていないかを生成後に確認する

## 3. `file:` 環境では `localStorage` を信用しすぎない

- `file:` や iframe、opaque origin では `localStorage` が例外になることがある
- セーブ・設定・ライセンス同意の読み書きで例外が出ると、画面描画とは別の箇所で不安定化する

### 教訓

- `localStorage.getItem/setItem/removeItem` は必ず `try/except` 相当で包む
- 保存失敗は黙殺せず、ログか警告を残す

## 今後のデバッグ手順

### 黒画面が出たら、まず以下を分けて考える

1. シーン遷移は起きているか
2. 画面切り替えは完了しているか
3. ゲーム画面の背景・UI・文字のどれが見えていないか
4. `file:` 制約由来の別エラーが混ざっていないか

### 最初に見るべきもの

- Console に `Loading scene: ...` が出ているか
- `#title-screen` に `active` が残っていないか
- `#title-screen.hidden` または `hidden=true` になっているか
- `#game-screen` が `active` か
- `#text-box`, `#ui-layer`, `#background-layer` の `display`, `visibility`, `opacity`, `z-index`

## 再発防止チェックリスト

- 画面切り替えは `class` だけでなく `hidden` も更新しているか
- 単一HTML出力物に相対URLが残っていないか
- `file:` 実行時に `localStorage` 例外で止まらないか
- Console の補助エラーに引っ張られず、まず「描画ロジック」と「画面切替」を分離して確認したか

## まとめ

- 今回の黒画面問題では、シーン読み込み失敗を疑うより先に、「タイトル画面が本当に消えているか」「単一HTMLとして不純物が残っていないか」を確認すべきだった
- 見た目の黒画面は、ロジック停止とは限らない
- 全画面UIの切替は、曖昧な透明化ではなく、明示的な非表示で管理する
